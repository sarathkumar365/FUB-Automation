# Workflow Builder UI — "Storyboard" Design & Phased Implementation Plan

> **Status:** Proposal · supersedes `workflow-builder-ui-plan.md` (which described a generic React-Flow MVP). This plan picks a distinct design tailored to this platform and engine, and lays out a phased implementation with first-class observability and traceability.

---

## 0. Why this plan exists

The Operator UI ([phase-2-implementation.md](./phase-2-implementation.md)) deliberately deferred the visual builder and shipped JSON-textarea CRUD. That bought time, but JSON authoring will not scale once non-engineers (ops leads, account managers) need to build or tweak SLA policies. We need a real authoring surface — but most "workflow builders" on the market (n8n, Make, Zapier, generic React-Flow clones) are wrong for this product:

- **n8n / Make**: freeform-canvas integration builders. Optimized for "connect 50 SaaS apps", not for "express a 5-scene SLA policy clearly." Their pain is well-documented: ["limited big-picture visibility for complex workflows"](https://www.flowmondo.com/article/n8n-vs-zapier-vs-make).
- **Zapier**: linear list, hides branching. Too thin for our branching SLAs.
- **Generic React-Flow**: lets users place nodes anywhere → "I rearranged it and now it's a mess" is a daily debug fire.
- **Temporal's critique** of graph editors ([blog](https://temporal.io/blog/the-fallacy-of-the-graph-why-your-next-workflow-should-be-code-not-a-diagram)) lands hard: "real logic hides in nodes, the diagram is a misleading parallel artifact." We must defeat both failures.

This plan proposes a **Storyboard** — a constrained, time-flowing, brand-native canvas where the diagram and the run trace are the same surface, scene cards expose the actual rule (not just the type), and the underlying state model is so well-instrumented that any bug is one keystroke from a reproducible trace.

## 1. The Storyboard concept (the "what" before the "how")

### 1.1 Visual identity is already a timeline

The landing page ([`ui/src/modules/landing/ui/LandingPage.tsx`](../../../../ui/src/modules/landing/ui/LandingPage.tsx)) renders a curved SVG spine with milestone cards positioned along it — that *is* this product's brand language. The builder should feel like you stepped *into* the landing page, not into a foreign canvas tool. Concretely:

- A **horizontally-flowing spine** (left → right) following a gentle bezier curve in the brand teal gradient (`#67e8f9 → #2dd4bf`), echoing the dashed `landing-path`.
- **Scene cards** float along the spine, styled like `landing-milestone` cards: white surface, `1px #dde6f2` border, `14px` radius, soft `rgba(15,23,42,0.04)` shadow, glass `backdrop-filter: blur(4px)`.
- **Manrope** for UI text, **JetBrains Mono** for IDs, expressions, and JSON — already the system fonts.
- Reuses tokens from [`ui/src/styles/tokens.css`](../../../../ui/src/styles/tokens.css) verbatim. Zero new color introductions.

### 1.2 Time-as-axis, not free-floating canvas

- Layout is **deterministic and computed** (Dagre LR rank). Users cannot drag scenes to arbitrary positions — they can only reorder, branch, and connect. This eliminates an entire class of layout-corruption bugs and "how do I get this back to neat" support tickets.
- **Optional Time-Stretch mode** scales horizontal distance proportionally to declared `delayMinutes` on `delay` / `wait_and_check_*` steps. A 10-min wait visibly takes more screen than a 0-min `set_variable`. SLA shape becomes legible at a glance.
- Branches fan **vertically** (above and below the spine) at result-code junctions, color-coded by semantic class (success / warning / error / neutral).

### 1.3 Scene cards expose the rule, not just the type

Every scene card surface shows the resolved-config preview, generated by a per-step-type "card formatter":

```
┌─ wait_and_check_claim ─────────── 5 min ──┐
│  Wait 5 minutes, then check if            │
│  the lead has been claimed.               │
│                                           │
│  ─── exits ───                            │
│  ● CLAIMED      → end (claimed)           │
│  ● NOT_CLAIMED  → reassign step           │
└───────────────────────────────────────────┘
```

The ASCII is illustrative — actual cards are HTML with chips, badges, and the brand typography. This directly counters Temporal's "diagram doesn't show the real control flow" critique.

### 1.4 One canvas, two modes: Compose / Trace

A segmented toggle in the header switches the *same* storyboard between:

- **Compose** — scenes are editable, palette is open, inspector shows config form.
- **Trace** — scenes overlay a real run's status (animated pulse for running, check for completed, red for failed, dimmed for skipped). The inspector shows the picked step's `resolved_config`, `outputs`, retry attempts, log entries.

The same component tree renders both. There is **no** separate "run viewer". This kills the "code drifted from diagram" failure mode permanently.

### 1.5 Three editing surfaces, one source of truth

Compose mode lets users flip between three surfaces with the same data:

1. **Storyboard** — the curved spine canvas (default).
2. **Outline** — a screenplay-style indented list. Scenes top-to-bottom, branches indented by result code. Pure keyboard-friendly. The fallback when the canvas is "too much."
3. **JSON** — raw graph JSON in a Monaco-style editor with schema-aware autocomplete. The escape hatch for power users.

All three views are pure derivations of one Zustand store. Switching is instant. Edits in any view propagate immediately.

### 1.6 Templating as first-class chips

JSONata template references (`{{ event.payload.leadId }}`) render as **clickable teal pills** (similar to Linear's @-mentions). In RJSF text fields, typing `{{` opens a typeahead pulling from `ExpressionScope`:
- `event.payload.*` (live from a sample event)
- `sourceLeadId`
- `steps.<previousNodeId>.outputs.*` (live from previous scenes' declared outputs)

Selecting a path inserts a chip. Hovering a chip shows the live value (when a sample event is provided). This defeats Temporal's "untyped global key-value store" critique.

### 1.7 Trigger as the opening scene

Rather than a special floating "trigger node," the trigger is the leftmost scene on the storyboard, styled distinctly (rounded-pill profile, brand-gradient border). It feels like Scene 0. Its config is edited in the same inspector that handles step scenes.

## 3. Engine semantics the UI must respect

These are non-negotiable invariants surfaced from `WorkflowExecutionManager.java` and `WorkflowStepExecutionService.java`. The builder must mirror them precisely; any deviation is a UI lie.

| Invariant | Implication for UI |
|---|---|
| **Single `entryNode`** (`WorkflowExecutionManager.materializeSteps`) | Storyboard has exactly one opening scene. UI prevents creating a second root. |
| **Result code → list of next node IDs (fan-out)** OR **`{terminal: REASON}`** (`WorkflowStepExecutionService.applyTransition`) | Each scene exit may either (a) connect to one or more next scenes, or (b) be a terminal pill carrying a reason string. Both are first-class card affordances. |
| **Implicit merge via `dependsOnNodeIds`** (computed at plan time from inverse edges) | When two arrows point at the same scene, the UI shows a small "joins N inputs" badge. No explicit merge node. |
| **Static vs dynamic result codes** (empty `declaredResultCodes()` = dynamic, e.g. `branch_on_field`) | UI reads `config.resultMapping` keys + `defaultResultCode` to derive exits. Editing the mapping prunes/adds edges with confirmation. |
| **Templates resolve at execute time** (raw config snapshot vs `resolved_config`) | Trace mode renders both the authored template *and* the resolved value side by side. Authored = source, resolved = audit. |
| **Retry is per-step-type with config override** (`RetryPolicy.fromMap`) | Inspector exposes a "Retry" sub-section with the effective policy. Default policy from catalog is shown as ghost values. |
| **Run statuses**: `PENDING / COMPLETED / FAILED / CANCELED / BLOCKED / DUPLICATE_IGNORED` | Trace mode uses these literal strings. Status colors come from `tokens.css` semantic palette. |
| **Step statuses**: `PENDING / WAITING_DEPENDENCY / PROCESSING / COMPLETED / FAILED / SKIPPED` | Each scene in Trace mode shows current status with brand-consistent badge + animation for `PROCESSING`. |
| **Failure reason codes**: `STEP_TYPE_NOT_FOUND`, `RUN_NOT_FOUND`, `EXECUTION_EXCEPTION`, `TRANSITION_NOT_DEFINED`, `WORKER_UNHANDLED_EXCEPTION`, `STALE_PROCESSING_TIMEOUT` | Trace mode and validation surface these literal codes in red badges with hover-tooltip explanations from a centralized `reasonCodes.ts`. |
| **Terminal race guard** (run finalized only if `status == PENDING`) | Trace mode handles "run already finalized while step processing" gracefully — shows skipped steps as such, not as bugs. |
| **Idempotency** (SHA256 of normalized key fields) | Builder doesn't expose this directly, but the dry-run UI shows the computed idempotency key for a sample event so authors understand re-trigger semantics. |

### 4.1 The kernel (Zustand store)

Single immer-backed store. Two key fields:

```ts
type BuilderState = {
  // canonical, sent to server
  graph: GraphV1            // { schemaVersion: 1, entryNode, nodes[] }
  trigger: TriggerSpec | null
  metadata: { key, name, description, status, versionNumber }

  // ephemeral, never sent
  layout: Record<NodeId, { x: number; y: number }>  // computed by layoutEngine
  selectedNodeId: NodeId | null
  selectedExitKey: { nodeId: NodeId; resultCode: string } | null
  mode: 'compose' | 'trace'
  surface: 'storyboard' | 'outline' | 'json'
  jsonBuffer: { text: string; parseError: string | null } | null
  validation: ValidationOverlay | null
  trace: { runId: number; stepStatuses: Record<NodeId, StepStatus> } | null
  dryRun: { sampleEvent: any; result: DryRunResponse | null } | null
}
```

All mutations go through **named action creators** in `actions.ts` (e.g., `addScene`, `connectExit`, `editConfig`, `setMode`, `loadGraph`). Action creators return `{ type, payload }` objects; an internal `dispatch(action)` function applies the immer recipe and pushes the action to `actionLog` with a timestamp + content hash before/after.

This gives us **time-travel**: `replay(stateAt(t))` reconstructs any prior state from the log.

### 4.2 Pure derivations

`selectors.ts` exports memoized derivations:

- `selectStoryboardModel(state)` → `{ scenes: SceneVM[], edges: ExitVM[] }` — feeds Storyboard surface.
- `selectOutlineModel(state)` → `OutlineNode[]` — feeds Outline surface.
- `selectJsonText(state)` → pretty-printed JSON for the JSON surface.
- `selectValidationByNode(state)` → `Record<NodeId, ValidationIssue[]>`.
- `selectAvailableScopePaths(state, currentNodeId)` → JSONata paths reachable from the selected scene (only ancestor steps' outputs are valid).

Each is a pure function. None of them mutate state. This makes the entire view layer trivially testable with hand-built state objects.

### 4.3 Two-way sync without loops

The traditional canvas-vs-JSON sync nightmare is solved by:

1. **The graph is canonical.** Storyboard and JSON are projections.
2. **JsonSurface owns its own buffer** while user is typing. Local controlled state. On every keystroke it attempts `zod.parse(GraphSchema, JSON.parse(text))`. If parse + schema both succeed, it dispatches `loadGraph(parsed)`. If either fails, the buffer stays local and a parse-error banner shows.
3. **When the store changes from elsewhere, JsonSurface re-derives its buffer** from `selectJsonText(state)` *only if* its own buffer's content hash matches the prior store hash. If the user has unparsed pending edits, the surface sticks to its own buffer until the user clicks "Discard local changes."
4. **Storyboard never owns layout in the store**. Layout is recomputed by `layoutEngine.layout(graph)` after every mutation. Drag is disabled. There is no race because there is no shared mutable state for positions.

This collapses the failure modes from "many" to "one explicit toggle."

## 5. Observability blueprint (the user's hardest requirement)

UIs of this complexity rot if you can't trace what happened. Every feature below is **table-stakes from Phase 0** — not "we'll add observability later." They are baked into the kernel.

### 5.1 Per-action structured log

Every action dispatched through `dispatch()`:

```ts
type LoggedAction = {
  ts: number              // performance.now() relative to page load
  wallTs: string          // ISO datetime
  type: string            // e.g. "addScene"
  payload: unknown
  hashBefore: string      // graph content hash
  hashAfter: string
  causedBy: string | null // upstream action type if synthetic
  correlationId: string | null  // if dispatched from an API response
}
```

Held in a 200-entry ring buffer. Visible in the Debug Overlay. Exportable as JSON ("Copy session log" button) — when an ops user reports a bug, they paste the log and we replay it.

### 5.2 Content-hash invariants

`contentHash.ts` produces a stable SHA1 of the canonical `graph` (key-sorted serialization). Computed:
- After every mutation (logged in the action).
- Before/after every save (round-trip integrity check).
- Compared between Storyboard's projection and JSON surface's projection on every render in dev — if they diverge, an in-app red banner fires (this catches selector bugs immediately).

### 5.3 Correlation IDs end-to-end

`correlationId.ts` generates a `nanoid(10)` per outgoing API call. Sent as `X-Correlation-Id` header. The Spring backend already accepts arbitrary headers and writes them to MDC via the existing webhook ingress filter — extend the same filter to honor the header for `/admin/*` paths. Then a single ID lets you grep:
- Browser console (logged on every fetch)
- Server logs (MDC tag)
- Server traces (span attribute)

### 5.4 OpenTelemetry web spans

`telemetry.ts` wraps the major workflows in spans:
- `builder.load(workflowKey)`
- `builder.validate(graphHash)`
- `builder.save(graphHash, op=create|update)`
- `builder.dryRun(graphHash, sampleHash)`
- `builder.modeChange(from, to)`

Spans are sent via OTLP/HTTP to the backend collector (already running for the engine). In dev they also render in the Debug Overlay timeline.

### 5.5 Debug overlay (Cmd+Shift+D)

A dev-only floating panel showing:
- Current store snapshot (collapsible JSON tree)
- Action log (last 50, click any to "rewind" — replays the store to that point)
- API call log (URL, status, duration, correlation ID, copy-curl button)
- Current `graph` content hash + view-projection hashes
- Current OTel span tree
- "Export session bundle" — zip of {graph, actionLog, apiLog, spans} with timestamps

In prod the overlay is gated behind `localStorage.setItem('builderDebug', '1')` so support engineers can ask the user to enable it without a deploy.

### 5.6 DOM tagging for E2E + accessibility

Every Storyboard scene gets `data-scene-id={nodeId}` and `data-step-type={type}`. Every exit edge gets `data-exit="{from}:{resultCode}"`. Every Inspector field gets `data-config-path` (e.g., `data-config-path="delayMinutes"`). These triple as:
- Playwright selectors (no brittle text-based queries)
- Screenshot annotators in test failures
- Debug Overlay "highlight in canvas" feature

### 5.7 Validation traceability

When the server returns validation errors, each error is mapped to:
- A `nodeId` (from path-parsing or explicit `nodePath` field — see Phase 3 backend extension)
- A code from a centralized `reasonCodes.ts` lookup (description, severity, fix hint)
- A jump-to-issue button in the IssuesPanel that selects the scene + scrolls + opens the inspector to the offending field

### 5.8 Test pyramid (built per-phase, not at the end)

| Layer | Tool | Per-phase coverage rule |
|---|---|---|
| Pure functions (selectors, adapters, hash, layout) | Vitest | Every PR adds tests for any new pure function. Property-based for round-trip (graph → view → graph). |
| Components in isolation | Vitest + Testing Library | Every new component renders from a hand-built state object. |
| Surface integration (Storyboard mounts a real graph) | Vitest with jsdom + a stubbed dagre | At least one test per surface per phase. |
| End-to-end (real backend via Testcontainers) | Playwright | Each phase ships at least one happy-path E2E that exercises the new capability. |
| Visual regression | Playwright screenshots | Snapshots of Storyboard at known graph fixtures. Fail on any diff > 0.1%. |
| Contract (FE ↔ BE schemas) | zod + sample fixtures committed to repo | Every catalog/run/workflow shape has a zod schema; fixture goes in `ui/src/__fixtures__/` and a test parses it. |

## 6. Phased implementation

> **Sequencing principle:** each phase ships a real, demoable, *useful* slice end-to-end (route is reachable, observability is wired). Phases are NOT "front-end first then back-end" or "scaffold then features" — that's how the JSON-textarea workaround happened the first time.

### Phase 0 — Foundations (1 PR, ~2 days)

**Scope:** Lay the kernel without any visible builder yet. Establishes contracts and observability hooks so all later work plugs into the same instrumentation.

**Deliverables:**
- New module `ui/src/modules/workflows-builder/` with the full directory layout in §4.
- Add deps to `ui/package.json`: `@xyflow/react@^12` (used only for SVG primitives + edge math, not as a full canvas), `@rjsf/core`, `@rjsf/utils`, `@rjsf/validator-ajv8`, `zustand`, `immer`, `dagrejs/dagre@^1`, `nanoid`, `@opentelemetry/api`, `@opentelemetry/sdk-trace-web`, `@opentelemetry/exporter-trace-otlp-http`.

- New routes `/admin/workflows/new` and `/admin/workflows/:key/edit` mounted to a placeholder page that just displays "Phase 0 ready" + the Debug Overlay.

### Phase 2 — Authoring primitives (3 PRs, ~5 days)

**Scope:** Make scenes editable. Add palette, drag/click-to-add, scene selection, scene deletion, exit connection, RJSF inspector. JSON surface comes online too — this is when the kernel earns its complexity.

**Deliverables:**
- `panels/Palette.tsx` — step-type catalog rendered as a vertical list of "drag me" cards, populated by `useStepTypesQuery`. Each card shows icon, displayName, declaredResultCodes preview.
- `panels/Inspector.tsx` — RJSF form bound to selected scene's `configSchema`. Uses Tailwind-themed RJSF templates from a new `panels/inspector/templates/` directory wrapping `ui/src/shared/ui/` primitives.
- Connection model: clicking an exit dot enters "connecting" mode; clicking a target scene completes the edge. ESC cancels. UI prevents creating cycles or undeclared result codes.
- Action creators: `addScene`, `removeScene`, `setEntryNode`, `editConfig`, `connectExit`, `disconnectExit`, `setTerminalExit`, `setExitTarget`.

### Phase 5 — Dry-run + advanced authoring (3 PRs, ~5 days)

**Scope:** Add dry-run capability (this requires backend work) and the polish features that make the builder feel professional: time-stretch, undo/redo, sample event editor, validation polishes.

**Deliverables — backend:**
- New package `service/workflow/dryrun/`:
  - `DryRunWorkflowRunner` — mirrors `WorkflowExecutionManager.materializeSteps` + `WorkflowStepExecutionService.applyTransition` traversal logic but in-memory only. Produces an ordered trace `List<DryRunTraceEntry>`.
  - `DryRunStepAdapter` interface with `simulate(config, scope, overrides)` returning `SimulatedResult { resultCode, outputs, logLines }`.

## 7. Risks and mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| Storyboard auto-layout produces visually awkward results for some graph shapes | Med | Dagre with carefully tuned `nodesep`/`ranksep` plus a Storybook gallery of fixture graphs (linear, branching, fan-out, deep, wide) tested visually each PR. Reserve a "Compact / Comfortable" density toggle as escape hatch. |
| Dry-run divergence from real engine semantics (silent lies) | High | Startup guard + a `WorkflowParityTest` that runs the same graph through real engine and dry-run with deterministic stubs and asserts identical trace structure. Drift fails CI. |
| Two-way JSON sync regressions | Med | Hash-divergence in-app banner in dev, surface-projection hash assertions in test, fuzz tests that hammer the JsonSurface with malformed input and assert kernel state never corrupts. |
| RJSF + Tailwind 4 theming clash | Low | Phase 2 builds custom field templates wrapping `ui/src/shared/ui/` primitives. No reliance on RJSF default CSS. |
| `branch_on_field` dynamic exits desync edges | Med | Dedicated action `editBranchMapping` that prunes edges whose result codes no longer exist (with confirm toast, undoable). Tests cover add/remove/rename mappings. |
| OpenTelemetry adds startup cost / bundle weight | Low | Lazy-load the OTel SDK; for prod, only ship the small `@opentelemetry/api` shim unless `localStorage.builderTelemetry=1` is set or NODE_ENV=development. |
| Engine adds a new step type, builder forgets to handle it | Med | `cardFormatters` registry has a fallback that renders `displayName + raw config JSON`; a Storybook story for "unknown step type" is required. Backend startup guard from Phase 5 ensures dry-run adapter is present. |
| Server validation error format changes | Low | All parsing goes through `validation/parseErrors.ts` with a versioned response shape; backend record `ValidateWorkflowResponse` is owned alongside the FE adapter. |
| User builds a 50-scene workflow and storyboard becomes slow | Low | Virtualize off-screen scenes (React Flow v12 supports viewport culling). Set a soft warning at 30 scenes suggesting a sub-workflow refactor (future feature). |

## 8. What this plan deliberately does NOT do (yet)

- **Sub-workflows / reusable fragments** — defer to a Phase 7 once we see real workflows >20 scenes.
- **Approval / review workflow on publish** — the existing lifecycle (DRAFT → ACTIVE) is enough for the current ops team; multi-person review gating is a product decision deferred.
- **Multi-trigger workflows** — engine model is one trigger per workflow; not changing.
- **Real-time multi-user collaboration** — out of scope; optimistic-lock errors handled gracefully but no live cursors.
- **Mobile authoring** — desktop-first; mobile is read-only Trace mode (existing responsive shell already handles this).

## 9. Verification plan (the proof we shipped it right)

**Engine parity:** rebuild `ASSIGNMENT_FOLLOWUP_SLA_V1` from scratch in the builder, save, activate, fire a synthetic trigger via the existing webhook ingress, and confirm the resulting `WorkflowRunEntity` and step trace are byte-identical to a run produced from the hand-written JSON workflow. Codified as a Testcontainers integration test.

**Layout determinism:** every fixture graph in `ui/src/__fixtures__/graphs/` produces the same layout twice. Screenshot baselines committed.

**Round-trip stability:** `graph → adapters → store → adapters → graph` is identity, hashed and asserted on every PR.

**Observability self-test:** every E2E test asserts that at least one `correlationId` round-tripped browser → server → browser successfully. Bug bounty: a scripted "monkey" test (fast-check property test that dispatches random valid actions) runs for 60s and asserts no view-projection-hash divergences fired.

**Performance budget:**
- First storyboard render < 200ms for graphs ≤ 20 scenes.
- Action dispatch + re-render < 16ms for the same.
- Save round-trip < 500ms p95 in local network.
- Bundle size of `workflows-builder` chunk ≤ 300KB gzip (lazy-loaded behind route).
